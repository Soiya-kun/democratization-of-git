# 01 コンセプト/用語

## 1. コンセプト

内部実装はGitを利用するが、ユーザー(非エンジニア)にはGit概念を極力見せず、分かりやすい言葉と導線で「作業の履歴を安全に残す」体験を提供する。

## 2. 対象ユーザー/前提

- 対象: エンジニアではないユーザー
- 前提: `git/branch/commit/stage` などの概念は難解であるため、UIでは使用しない
- 前提: ユーザーが自分でGitコマンドを実行しない (履歴操作は本アプリが担う)
- 例外: 開発者向けログ/デバッグ表示には内部用語が出てもよい (UIには出さない)

## 3. 初回起動の初期設定

- 初回起動時にユーザーへ以下を尋ね、アプリ設定として保存する
  - 履歴に記録する「名前」(Git `user.name` 相当)
  - 履歴に記録する「メールアドレス」(Git `user.email` 相当)
- 本アプリが「管理するプロジェクト」に対してのみ、この設定を使用する
  - 対象: このアプリで扱うプロジェクト (Gitリポジトリで `main` が存在すること)
  - 既存リポジトリでも `main` があれば扱える
- 設定が未入力の場合は、プロジェクト開始/履歴保存系の操作の前に入力を促し、入力されるまで実行しない

## 4. 用語 (UI表示) と内部実装 (Git) の対応

| UIでの呼称 | ユーザーに伝える意味 | 内部の実態 (Git) |
|---|---|---|
| ワーク(作業) | いま取り組んでいる作業の単位 | ブランチ(アプリ管理) |
| 作業名 | ワークの名前 | ブランチ名(規則により生成) |
| 変更 | ファイルが変わった状態 | working tree の差分 |
| 変更内容 | 何が変わったか | diff |
| 次の履歴に含める | 次回の保存対象にする | stage (`git add`) |
| 次の履歴に含めない | 保存対象から外す | unstage (`git reset ... -- <file>`) |
| 変更を取り消す | 変更を元に戻す | checkout/restore, clean |
| この修正の履歴を保存する | 今の状態を履歴として残す | commit |
| 作業を完了する | 作業を完成版に反映して終了する | mainへmerge → mainへ切替 → 作業ブランチ削除 |
| クラウドへ保存(将来) | 別PCでも使えるようにする | push |

UI文言は上記を基本とし、Git用語はUI上に出さない。
